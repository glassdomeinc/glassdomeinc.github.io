Discussion about Go Language and Project Structure

November 1, 2022 Â· 5 min read
Mike
Software Engineer Intern

This is a discussion about the directory structure of Go applications.

Introduction to Goâ€‹

I got to know Go on the 2015 podcast "I'm a Programmer" episode "The Go Language Attraction of Google Engineers". Yeom Jae-hyun, who worked as a Google engineer, appeared on the show and talked about the Go language while publishing a book called "Discovery Go". And I did a little tutorial through a Go study group, but I haven't actually used Go in practice since.

Glassdome uses Go language to develop applications.

The Benefits of Goâ€‹

First off, Go is quite popular in the industry. It's a language that I believe will continue to be popular.

The syntax resembles C, which is learned in basically any school.

At the same time, it is easier to use than C. (obviously)

Furthermore, it performs better than VM or script engine-based languages.

Another advantage that I found was,

Go applications are statically compiled, minimizing runtime dependence, and are very small when built with a container image.

Empty container without even an OS can only be run by inserting the Go binary. Therefore, it takes less time to upload images when developing and deploying, and less time to download container images from the Kubernetes node and less time to prepare for execution.

Go Container Image Size

When you build a Web application server with JVM or NodeJS, no matter how small you try to make it, it can easily exceed 100MBs and even become hundreds of MBs. On the other hand, you can create a Go container of less than 10 MB even if you include all the packages you need.

Install Goâ€‹

tip

We recommend that you install it as a GVM.

gvm install go1.19

Go Project Structureâ€‹

Because Go syntax is easy, developers who have used other programming languages can adapt quickly.

But sometimes when looking at the code you might not know what to do.

For example, how do you capture the directory structure of the Go project?

Today I'm going to talk about this.

info

The document you want to refer to is Standard Go Project Layout.

The Go Project is a Moduleâ€‹

Since Go 1.14, Go has had the the concept of a Module.

Before the Go Module came out, Go had a structure where dependency packages were gathered and installed in the same place the execution environment was installed (GOPATH); which is similar to Python.

info

Similarly, if you do not use venv in Python the package is installed in the python installation path python-x.y.z/site-packages/.

But we don't need to go over old information, you can consider all the latest Go releases as using Go Modules.

Dependent packages are managed separately for each module.

info

Go Module = Project (in other languages)

Getting Started with Go Moduleâ€‹

Let's make a module called mymodule.

mkdir mymodule
cd mymodule

go mod init github.com/glassdomeinc/mymodule

The file go.mod is created.

go.mod
module github.com/glassdomeinc/mymodule

go 1.19

No Separate src Directory, module is root packageâ€‹

In JVM languages, including Java, the path that the compiler recognizes as a source code file is conventionally defined as src/main/java.

But instead of having that rule in Go, the Go module directory itself becomes the root package for that module.

If you look at many Go open source projects hosted on GitHub, you can see that the repository itself is a Go Module that contains the source code *.go directly in the root directory.

+ mymodule/
+-- go.mod
+-- go.sum
+-- README.md
+-- myfile.go

You can also put the go.mod and source code *.go files side by side in the module root directory.

myfile.go
package mymodule

func Xxx() {
  ...
}

The main package (entry point code) is located under the cmd/ directoryâ€‹

main.go
package main

func main() {
  ...
}

The entry point code for the Go application is written in a reserved package called main. However, the main package cannot be placed in the same directory as any other generic package. If you place main.go in the root path of the module, it will take away the root package digits (most importantly) where the other source code files will fit.

Therefore, main.go is placed in the cmd/ directory instead of the module's root path.

+ mymodule/
+-+ cmd/
| +-- main.go
+-- go.mod
+-- go.sum
+-- README.md
+-- myfile.go

(We don't really execute this well) If there are multiple entry points, place them in each subdirectory under cmd/.

caution

There can only be 0~1 main in each directory

+ mymodule/
+-+ cmd/
| +-+ main1
| | +-- main.go
| +-+ main2
|   +-- main.go
+-- go.mod
+-- go.sum
+-- README.md
+-- myfile.go
caution
