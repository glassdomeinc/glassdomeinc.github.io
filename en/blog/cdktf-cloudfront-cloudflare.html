Deploying an S3 static website with CDK for Terraform on Cloudfront vs Cloudflare

November 8, 2022 · 7 min read
Adil
Software Engineer

This tutorial shows what I did to setup and deploy an example static website on Cloudflare and Amazon Cloudfront using Terraform (CDK, Typescript)

Cloudflare and Cloudfront are similar CDN, DNS and load balancing providers that can help host your API or website and make it more available. Despite their similarity, setting them up with an S3 Hosted static website is obviously different.

In this tutorial, I'll give some code snippets and pointers on how to set up the bare working minimum for each. The rest like more secure S3 Policies, hosting directly on Cloudflare or using an on-premise hosted website instead of an S3 Bucket will be up to the reader.

Prerequisites​

Make sure to have cdktf, terraform, node, npm, react installed on your system.

'web' folder can be initialized using:

$ cd lib
$ npm i -g create-react-app
$ create-react-app web
$ cd web
$ npm run build

The root folder can be initialized with:

$ cdktf init

The rest needs to be created.


Boilerplate code to start with​

First, get provider dependencies by running:

$ cdktf get

The dependencies will be store in the directory '.gen' under root folder of the project.

Let's now create our main -- index.ts in the root folder, from which the Terraform app can run:​

index.ts
import * as dotenv from 'dotenv'
import { App } from 'cdktf'
import { TestStack } from './lib/test'

dotenv.config()

const app = new App()

new TestStack(app, 'test')

app.synth()

And then the index.ts in 'lib' where we define our Terraform Stack:​

First, the dependencies are as such:

/lib/index.ts
import { TerraformStack } from 'cdktf'
import { Construct } from 'constructs'
import { MyBackend } from '../backend'
import { MyProvider } from '../provider'
import { CloudflareTest } from './cloudflare-test'
import { CloudfrontTest } from './cloudfront-test'
import { CloudflareProvider } from '../../.gen/providers/cloudflare/provider'

export interface TestStackProps {
  region?: string
}

We define several providers (this can also be done in providers.ts) because we use AWS S3 to host our files and Cloudflare as a standalone provider for DNS and CDN:

/lib/index.ts
export class TestStack extends TerraformStack {
  constructor(scope: Construct, id: string, props?: TestStackProps) {
    super(scope, id)

    // defining providers
    new AwsProvider(this, 'provider', {
      profile: process.env.AWS_PROFILE,
      region: props?.region,
    })
    new CloudflareProvider(this, 'cloudflare-provider', {
      apiToken: '<YOUR_API_TOKEN>',
    })

    // enabling backend
    MyBackend.of(this, id)

    // ---

    const acmCertificateArn = '<YOUR ACM CERTIFICATE ARN>'

    new CloudfrontTest(this, 'adil-s3-test', {
      domain: '<YOUR PURCHASED DOMAIN NAME>',
      hostedZoneId: '<YOUR HOSTED ZONE ID>',
      acmCertificateArn,
    })

    new CloudflareTest(this, 'cloudflare-test', {
      domain: '<YOUR PURCHASED DOMAIN NAME>',
    })
  }
}
  • To get some of the credentials and other values like \<YOUR_API_TOKEN>, refer to the Using Cloudflare section down below

Setting up S3 bucket hosting​

Here we have to make sure our bucket is reachable, thus we set its ACL rule as 'public-read' and security policy to allow PublicReadGetObject as in the code snippet below:

/lib/cloudflare-test.ts
const s3_bucket = new S3Bucket(this, 's3-bucket', {
  bucket: props.domain,
})

new S3BucketAcl(this, 's3-bucket-acl', {
  bucket: s3_bucket.bucket,
  acl: 'public-read',
})

new S3BucketPolicy(this, 's3-bucket-policy', {
  bucket: s3_bucket.bucket,
  policy: `{
        "Version": "2012-10-17",
        "Statement": [
          {
            "Sid": "PublicReadGetObject",
            "Effect": "Allow",
            "Principal": "*",
            "Action": [
              "s3:GetObject"
            ],
            "Resource": [
              "arn:aws:s3:::${props.domain}/*"
            ]
          }
        ]
      }`,
})

// Static Website Hosting
const website_сonfiguration = new S3BucketWebsiteConfiguration(this, 's3-website-configuration', {
  bucket: s3_bucket.bucket,
  indexDocument: { suffix: 'index.html' },
  errorDocument: { key: 'index.html' },
})

Then, we enable website hosting. Note, in older versions of hashicorp/aws provider all of these configurations can be done using just the 'S3Bucket' object. In the latest versions, ACL, policy and other settings for the bucket have to be defined separately.


Uploading static website files to S3 as S3 Objects​

/lib/cloudflare-test.ts
// Get all the files from build folder, skip directories
const files = glob.sync('./lib/web/build/**/*', { absolute: false, nodir: true })

// Create bucket object for each file
for (const file of files) {
  new S3Object(this, `aws_s3_bucket_object_${path.basename(file)}`, {
    dependsOn: [s3_bucket], // Wait untill the bucket is not created
    key: file.replace(`./lib/web/build/`, ''), // Using relative path for folder structure on S3
    bucket: props.domain,
    source: path.resolve(file), // Using absolute path to upload
    etag: `${Date.now()}`,
    contentType: mime.contentType(path.extname(file)) || undefined, // Set the content-type for each object
  })
}

*Note: Here a dependsOn parameter is used to wait for the bucket to be created before starting upload. Also, make sure to change the Terraform name of each S3Object accordingly.


Adding Cloudflare CDN Records​

Then, we add a CNAME record to Cloudflare pointing to that S3 Bucket URL that is now publicly available (and www just in case):

/lib/cloudflare-test.ts
new CloudflareRecord(this, 'cloudflare-record-naked', {
  zoneId: '2cdf31b7a1cd506ef0307f912181bfdc',
  name: props.domain,
  value: website_сonfiguration.websiteEndpoint,
  type: 'CNAME',
  ttl: 1,
  proxied: true,
})

new CloudflareRecord(this, 'cloudflare-record-www', {
  zoneId: '2cdf31b7a1cd506ef0307f912181bfdc',
  name: 'www',
  value: props.domain,
  type: 'CNAME',
  ttl: 1,
  proxied: true,
})

That's it. After running:

$ cdktf deploy

The static website should now be available via your purchased domain


Using Amazon CloudFront​

Setting up an S3 Bucket​


Before we configure the bucket, we need Cloudfront Origin Access Identity to then use that in our bucket's IAM policy configuration:

/lib/cloudfront-test.ts
const cloudfront_origin_access_identity = new CloudfrontOriginAccessIdentity(
  this,
  'cloudfront-origin-access-identity',
  { comment: props.domain }
)

This time the bucket can be made private and without website configuration:

/lib/cloudfront-test.ts
const s3_bucket = new S3Bucket(this, 's3-bucket', {
  bucket: props.domain,
})

new S3BucketAcl(this, 's3-bucket-acl', {
  bucket: s3_bucket.bucket,
  acl: 'private',
})

new S3BucketPolicy(this, 's3-bucket-policy', {
  bucket: s3_bucket.bucket,
  policy: `{
        "Version": "2012-10-17",
        "Statement": [
          {
            "Sid": "PublicReadGetObject",
            "Effect": "Allow",
            "Principal": {
              "AWS": "arn:aws:iam::cloudfront:user/CloudFront Origin Access Identity ${cloudfront_origin_access_identity.id}"
            },
            "Action": [
              "s3:GetObject"
            ],
            "Resource": [
              "arn:aws:s3:::${props.domain}/*"
            ]
          }
        ]
      }`,
})

Then, we add objects to the bucket in the same manner as with Cloudflare

After which, the Cloudfront distribution is configured as such:

/lib/cloudfront-test.ts
const cloudfront_distribution = new CloudfrontDistribution(this, `aws_cloudfront_${props.domain}`, {
  enabled: true,
  dependsOn: [s3_bucket],
  defaultRootObject: 'index.html',
  aliases: [props.domain],
  customErrorResponse: [
    {
      errorCode: 404,
      responseCode: 200,
      responsePagePath: '/index.html',
    },
  ],
  origin: [
    {
      originId: originId,
      domainName: s3_bucket.bucketDomainName,
      s3OriginConfig: {
        originAccessIdentity: cloudfront_origin_access_identity.cloudfrontAccessIdentityPath,
      },
    },
  ],
  defaultCacheBehavior: {
    allowedMethods: ['GET', 'HEAD'],
    cachedMethods: ['GET', 'HEAD'],
    forwardedValues: {
      cookies: { forward: 'none' },
      queryString: false,
    },
    targetOriginId: originId,
    viewerProtocolPolicy: 'allow-all',
  },
  restrictions: {
    geoRestriction: {
      restrictionType: 'none',
    },
  },
  viewerCertificate: {
    // *.glassdome.cloud 인증서
    acmCertificateArn: props.acmCertificateArn,
    minimumProtocolVersion: 'TLSv1.2_2021',
    sslSupportMethod: 'sni-only',
  },
})
  • Note: It's important to add 'aliases' field for your domain to work.

After that, we need to create an A Record in Route 53 as such:

/lib/cloudfront-test.ts
new Route53Record(this, 'route53-record', {
  zoneId: props.hostedZoneId,
  type: 'A',
  name: props.domain,
  alias: [
    {
      name: cloudfront_distribution.domainName,
      zoneId: cloudfront_distribution.hostedZoneId,
      evaluateTargetHealth: false,
    },
  ],
})

This will ensure the domain name works and points to the correct content.

Conclusion​

As you might have noticed, the Cloudfront is slightly easier to set-up safely without exposing S3 Bucket URL to the public. However, Cloudflare might offer other additional safety and performance benefits after careful set up. For this tutorial, we only focused on the bare working minimum, so that'll do for now.
